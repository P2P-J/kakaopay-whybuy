# 05-SUBMISSION-OUTLINE — 제출 글 뼈대 (왜샀지)

> 용도: 예선 제출 글을 쓸 때의 설계도. 각 섹션에 [핵심 메시지] / [소재 출처] / [한 줄 초안]을 둔다. 소재 출처의 "결정 #n"은 시나리오 문서 9번 섹션의 결정 기록, "시나리오 n"은 시나리오 문서의 섹션 번호.

---

## 0. 제목 + 훅 (3줄 이내)

- [핵심 메시지] 초보 투자자가 가장 많이 하는 혼잣말을 그대로 서비스 이름으로.
- [한 줄 초안] **"이 주식 왜 샀지?" — 대한민국 초보 투자자가 가장 많이 하는 말입니다. 우리는 이 혼잣말에 답하는 플러그인을 만들었습니다: 왜샀지(whybuy).**
- [소재] 결정 #7 (명명), 시나리오 0번 섹션

## 1. 문제 — 초보 투자자는 어디서 실제로 돈을 잃는가

- [핵심 메시지] 손실의 원인은 "뭘 살지 몰라서"가 아니라 "팔 때 근거가 없어서". 학술적 이름: 처분효과.
- [구성] ①센터장 인터뷰의 출제 의도 인용 ("내가 손해 보지 않았다라든지... 어떻게 도움을 받을 수 있을까") ②20만 명 실증: 이익 종목 매도 확률 2배, 경험 부족할수록 심함, "논리적 근거 없이"가 정의에 포함 ③매도 판단의 유일한 기준이 '매수가 대비 손익률'이라는 뿌리
- [소재] 시나리오 1번 섹션

## 2. 허를 찌른 지점 — 남들이 만들 것과 만들 수 없는 것

- [핵심 메시지] 매매 추천 봇은 이중 함정(규제상 출시 불가 + "수학적 최적점을 찾는 문제가 아니다"라는 출제 의도). 정보 요약 챗봇은 토스가 1년 앞선 축에서의 추격전. 우리는 축을 바꿨다 — **시장의 시간이 아니라 나의 시간.**
- [구성] 함정 2단 → "기준점의 틈새" 선언 → 토스 비교 표 (초보자의 질문 6개 중 아래 3개에 토스는 답이 없다)
- [소재] 시나리오 6번 섹션 (토스 정면 비교), 결정 #1 (허 찌르기 기조)

## 3. 솔루션 — 세 개의 질문, 세 개의 스킬

- [핵심 메시지] 초보자가 실제로 하는 질문 순서가 곧 제품 구조다.
- [구성] 표 하나로:
  - "내가 산 뒤로 뭐가 달라졌어?" → buy-timeline (매수일 앵커 + 수익률·시장 대비 + 변동–공시 매칭)
  - "나 이거 왜 샀더라?" → reason-recall (객관식 재인 — 주관식을 못 푸는 사람도 보기는 알아본다)
  - "그때 그 이유, 아직 유효해?" → thesis-audit (공시 원문 대조, 유효/약화/반증/확인불가)
- [데모 한 장면 삽입] "-N%인데, 공시된 사실 중 나빠진 것은 0건입니다" 브리핑 캡처
- [소재] 시나리오 3번 섹션, 04-DEMO 막 1

## 4. 이름 이야기 — 어땠지 옆의 왜샀지 (독립 수렴)

- [핵심 메시지] 우리는 '어땠지'의 존재를 모른 채 사용자의 실제 말버릇에서 이 이름을 지었고, 나중에 카카오페이증권 제품군과 정합함을 발견했다. **사용자 언어에서 출발하면 회사의 제품 언어에 도달한다는 방증.**
- [소재] 결정 #7, 시나리오 0번 섹션 명명 기원 — 이 문단은 짧게, 그러나 반드시 넣는다 (심사위원 기억에 남는 스토리)

## 5. 설득의 과정을 아키텍처로 — 평가 기준에 대한 정면 응답

- [핵심 메시지] 심사 기준이 "설득해 나가는 과정의 설계"였다. 우리의 답: 근거→납득→안심→실행을 코드 구조로 옮겼다.
- [구성] 4원칙 (원문만 인용 / 권유 금지 / **부재의 정직** / 판정 근거 공개) + 각각의 기계 강제 장치 (rcp_no 링크 필수, 컴플라이언스 게이트, "공식 기록상 없음" 정상 경로, rule_id) — "규제 준수를 문서의 약속이 아니라 코드로 증명"
- [출제사 코멘트 인용] "규제 준수까지 정확성과 근거가 검증되는 AI를 함께 풀어갈 인재" → 게이트 로그 캡처로 응답
- [소재] 시나리오 4번 섹션, PRD 7번 섹션

## 6. 만든 과정 — 해답을 만들어 나간 길 (결정 기록 발췌)

- [핵심 메시지] 센터장: "특정 해답보다 그 해답을 만들어 나가는 과정을 어떻게 설계했느냐를 본다" → 우리의 과정 자체를 증거로 제출.
- [구성] 결정 기록에서 서사가 있는 4개만 발췌, 시간순:
  1. **비자명 발굴** (결정 #1): 명백한 답 대신 행동경제학 실증 + 규제 구조의 교차점에서 문제 재정의
  2. **핵심 반전** (결정 #3): "초보자는 산 이유도 모른다"는 지적이 1차 컨셉을 뒤집고 → 재인(객관식) 방식과 매수일 앵커를 낳음. 그리고 이 UX 결정이 뜻밖에 판정 엔진의 성립 조건이 됨 (유형이 유한해져 룰 판정이 가능해짐)
  3. **경쟁 기능의 변형 수용** (결정 #5): 토스 AI 시그널의 좋은 점을 가져오되 뉴스를 버리고 "원인의 유무 판정"으로 변형 — 빈손("공시상 없음")이 곧 가치가 되는 설계
  4. **요구의 변형 수용** (결정 #11): "좋음/나쁨 태그" 요구를 규제 선과 충돌 없는 "시장 대비 상회/하회"(산술 비교)로 재설계
- [소재] 시나리오 9번 결정 기록 #1, #3, #5, #11

## 7. 신뢰의 증명 — 심사위원이 직접 확인할 수 있는 것

- [핵심 메시지] 믿어달라 하지 않는다. 세 가지를 직접 해보시라.
- [구성] ①브리핑의 [원문] 링크 클릭 → 실제 DART 원문 (전 케이스 실존 종목 — 카카오 포함, "출제사의 모기업 주식으로도 정직하게 돌렸다") ②게이트 로그 열람 ③재현 명령 한 줄 → 같은 판정 (결정성)
- [정직 고지 포함] 국내 전용(해외는 아직), 게이트 검출 범위, 실행 환경 제약 — 한계를 숨기지 않는 것도 신뢰 설계의 일부
- [소재] 04-DEMO 막 5, DEV-PLAN 부록 B, 결정 #10·#12

## 8. 로드맵 — 실무자의 Codex에서 A씨의 손끝까지

- [핵심 메시지] 오늘은 실무자 검증 도구, 내일은 앱 속 '어땠지' 옆자리, 종착점은 센터장이 말한 그 에이전트.
- [구성] 센터장 인터뷰 인용 ("굳이 앱이 있을까... 실행해 주는 애는 카카오페이증권의 에이전트일 거다") → 3단 로드맵 → 전환 비용 표 한 줄 요약: "비싼 교체는 전부 껍데기이고, 본질(판정 룰·게이트·시점 필터)은 한 줄도 다시 쓰지 않는다"
- [입력 2→0 논리] **"외부 도구인 오늘은 종목과 매수일 두 개를 입력받지만, 카카오페이증권 안에서는 이 둘이 원장에 이미 있으므로 입력이 0이 된다. 이 서비스를 증권사가 만들어야 하는 이유다."** — 부록 D의 계좌 연동 행(케이스=실사용자 보유 종목, 원장에 user 차원 추가)과 연결되는 논리
- [소재] 시나리오 2번 섹션 로드맵, DEV-PLAN 부록 D

## 9. 마무리 (2~3줄)

- [한 줄 초안] **왜샀지는 답을 주는 AI가 아닙니다. 초보 투자자가 스스로 납득하게 만드는 AI입니다 — 카카오페이증권이 평가 기준으로 말한 '설득의 과정', 그 자체를 만들었습니다.**

- 전체 분량: 플랫폼 규정에 맞추되, 섹션 4·6처럼 스토리성 있는 부분은 짧고 단단하게 — 길이보다 리듬
- 캡처 삽입 위치: 3번 섹션 (타임라인 브리핑), 5번 섹션 (게이트 로그), 7번 섹션 (DART 원문 열린 화면) — 글보다 화면이 증명하는 구조

- 금지: "혁신적", "최초"류의 수식어. 대신 "국내외 공백 확인 완료" 같은 검증된 표현 사용
