# 왜샀지 (whybuy) — 제출 글

> 예선 제출 원고. 뼈대는 `05-SUBMISSION-OUTLINE`. 모든 수치·동작은 저장소에서 재현 가능(README "검증 방법").

## "이 주식 왜 샀지?"

대한민국 초보 투자자가 가장 많이 하는 혼잣말입니다. 우리는 이 혼잣말에 답하는 플러그인을 만들었습니다 — **왜샀지(whybuy)**. 추천하지 않습니다. 판정합니다.

---

## 1. 문제 — 초보 투자자는 어디서 실제로 돈을 잃는가

손실의 원인은 "뭘 살지 몰라서"가 아닙니다. **팔 때 근거가 없어서**입니다.

센터장 인터뷰의 출제 의도가 그대로 문제입니다: *"초보 투자자들이 매수 매도를 할 때 내가 손해 보지 않았다라든지, 이 매수 매도를 어떻게 해야 될지 잘 모르겠는데 어떻게 도움을 받을 수 있을까."*

학술적 이름은 **처분효과**입니다. 자본시장연구원의 국내 개인투자자 약 20만 명 실증 분석:

- 매수가 대비 **상승한 주식을 매도할 확률이 하락한 주식의 2배 이상** — 이익은 조급하게 실현하고, 손실은 인정하지 못해 쥐고 있는다
- 이 효과는 **투자경험이 부족한 투자자**에서 현저하고, 강할수록 투자성과가 저조하다

뿌리는 하나입니다. 매도 판단의 유일한 기준이 **'매수가 대비 손익률'** 이라는 것. 빨간 숫자 하나만 보고 판다는 뜻입니다.

## 2. 허를 찌른 지점 — 만들 것과 만들 수 없는 것

**매매 추천 봇**은 이중 함정입니다. ①개인화된 매매 지시·단정적 판단은 투자자문업 규제의 정면이라 증권사가 출시할 수 없고 ②출제 의도도 "수학적 최적점을 찾는 문제가 아니다"입니다. **정보 요약 챗봇**은 이미 '어땠지'가 1년 앞선 축에서의 추격전입니다.

그래서 축을 바꿨습니다 — **시장의 시간이 아니라 나의 시간.** '어땠지'가 "오늘 시장 어땠지?"에 답한다면, 왜샀지는 "내가 산 뒤로 뭐가 달라졌지?"에 답합니다. 초보자가 실제로 하는 질문 중 "내가 산 뒤로 뭐가 달라졌어 / 나 이거 왜 샀더라 / 그 이유 아직 유효해"에는 기존 서비스가 답을 갖고 있지 않습니다.

## 3. 솔루션 — 세 개의 질문, 세 개의 스킬

초보자가 실제로 하는 질문 순서가 곧 제품 구조입니다.

| 초보자의 질문 | 스킬 | 하는 일 |
|---|---|---|
| "내가 산 뒤로 뭐가 달라졌어?" | **buy-timeline** | 매수일 앵커 타임라인 + 수익률·시장 대비 + 급변–공시 매칭 |
| "나 이거 왜 샀더라?" | **reason-recall** | 객관식 재인 — 주관식을 못 푸는 사람도 보기는 알아본다 |
| "그때 그 이유, 아직 유효해?" | **thesis-audit** | 공시 원문 대조 — 유효/약화/반증/확인불가 (+거울 모드) |

킬러 한 줄은 이것입니다: **"−11.4%인데, 공시된 사실 중 나빠진 것은 0건입니다."** 가격은 빠졌지만 산 이유는 공식 기록상 그대로라는 것 — 처분효과를 정면으로 반박하는 문장이고, 실데이터(삼성전자 case-001)에서 실제로 나옵니다.

> 【캡처 ①】 case-001 타임라인 브리핑 — "변한 것 / 변하지 않은 것 / 주가가 크게 움직인 날(공시 없음)"

## 4. 이름 이야기 — 어땠지 옆의 왜샀지

이 이름은 아엔이 지었습니다. 중요한 사실 하나 — **아엔은 카카오페이증권에 '어땠지'가 있다는 걸 모른 채, 초보 투자자들의 실제 말버릇에서 이 이름을 떠올렸습니다.** 어땠지와의 정합은 사후에 발견된 수렴입니다. 사용자의 실제 언어에서 출발하면 회사의 제품 언어에 도달한다 — 이 독립 수렴 자체가 방향이 맞았다는 방증입니다.

## 5. 설득의 과정을 아키텍처로 — 평가 기준에 대한 정면 응답

심사 기준은 "설득해 나가는 과정의 설계"였습니다. 우리의 답은 근거→납득→안심을 **코드 구조**로 옮긴 것입니다. 4원칙과 각각의 기계 강제 장치:

1. **공식 기록 원문만 인용** — 모든 판정에 rcp_no + DART 원문 URL. 뉴스·커뮤니티는 판정에 안 쓴다.
2. **해석·권유 금지, 기계로 강제** — 브리핑은 저장 직전 컴플라이언스 게이트를 통과한다. 지시·예측·평가어가 감지되면 차단. **약속이 아니라 테스트로 증명**했다: 음성 세트(권유 문장) 검출 100%, 양성 세트(사실 서술) 오탐 0.
3. **부재의 정직** — 없으면 없다고 말한다. 급변일에 매칭 공시가 없으면 설명을 지어내지 않고 "공식 기록상 변한 사실 없음"을 정상 출력한다.
4. **판정 근거 공개** — 같은 원장 + 같은 기준일 → 항상 같은 판정. 모든 판정에 rule_id.

출제사 공식 코멘트 — *"규제 준수까지 정확성과 근거가 검증되는 AI"* — 에 대한 응답이 이 게이트입니다. 규제 준수를 문서의 약속이 아니라 **코드로 증명**합니다.

**실제로 이렇게 동작했습니다(개발 실증):**
- 판정 엔진(`verdict_engine`)은 시세를 임포트하지 않는다 — 소스에 가격 관련 문자열 0건을 테스트로 강제(가격이 판정에 새지 않게).
- 시점 필터는 데이터 계층 한 곳에서 강제 — 도구가 늘어도 "그때 알 수 없던 미래 정보"가 새는 게 구조적으로 불가능.
- **KB금융 안전핀 실증**: 사유 라이브러리의 "오래 갖고 갈 우량 기업" 보기는 영업이익 계정이 유효할 때만 노출된다. KB금융은 금융지주라 주요계정에 영업이익이 없어(법인세차감전순이익만) 이 보기가 자동 배제됐다 — 실적 판정 룰이 엉뚱한 계정에 오작동하는 것을 실데이터가 막았다.

> 【캡처 ②】 `reports/case-001/gate-log-*.txt` — 권유 표현 0건·필수 요소 충족의 기계 검사 로그

## 6. 만든 과정 — 해답을 만들어 나간 길

센터장의 말: *"특정 해답보다 그 해답을 만들어 나가는 과정을 어떻게 설계했느냐를 본다."* 그래서 과정 자체를 증거로 냅니다. 결정 기록에서 서사가 있는 넷:

1. **비자명 발굴** — 명백한 답(추천 봇) 대신 행동경제학 실증 + 규제 구조의 교차점에서 "추천이 아니라 논거 검증"으로 문제를 재정의.
2. **핵심 반전** — "초보자는 산 이유도 모른다"는 지적이 1차 컨셉을 뒤집어 **재인(객관식) + 매수일 앵커**를 낳았다. 그리고 이 UX 결정이 뜻밖에 판정 엔진의 성립 조건이 됐다 — 논거를 객관식으로 받으니 유형이 유한해져 룰 판정이 가능해졌다.
3. **경쟁 기능의 변형 수용** — 앞선 서비스의 급변 설명 기능을 가져오되 뉴스를 버리고 "원인의 유무 판정"으로 변형. 빈손("공시상 없음")이 곧 가치가 되는 설계.
4. **요구의 변형 수용** — "좋음/나쁨 태그" 요구를 규제 선과 충돌 없는 "시장 대비 상회/유사/하회"(산술 비교)로 재설계. 계산 결과이지 판단이 아니다.

과정을 코드로 검증한 흔적: 게이트 4개 통과가 아니라 **G0부터 G9까지 10단계 게이트를 각각 완료판정(테스트+컨트랙트)으로 닫으며** 쌓았고, 최종 스위트 **151개 테스트가 통과**합니다.

## 7. 신뢰의 증명 — 직접 확인할 수 있는 것

믿어달라 하지 않습니다. 세 가지를 직접 해보실 수 있습니다.

1. **브리핑의 [원문] 링크 클릭 → 실제 DART 원문이 열립니다.** 전 케이스가 실존 종목입니다 — 카카오 포함, **출제사의 모기업 주식으로도 정직하게 돌렸습니다.** 산출물의 DART 링크 44개를 자동 검사기로 확인했고(200 응답 + 본문 오류 문구 부재 + 문서 요소 존재), 전부 유효했습니다.
2. **게이트 로그 열람** — 규제 준수가 기계로 찍힙니다.
3. **재현 명령 한 줄 → 같은 판정.** 결정적 연산이라 누가 돌려도 같은 답입니다.

**거울 장면이 카카오 실데이터로 성립합니다.** 사용자가 "실적 개선 기대"로 샀다고 기록한 논거가, 기준일 시점 카카오가 스스로 제출한 공시(20251Q 영업이익 전년 동기 대비 −12%)로 **반증**됩니다. 같은 시점 "당신이 고르지 않았지만 알아야 할 새로운 사실"에는 감자결정이 자동으로 잡힙니다 — 중대 유형 ∧ 매수일 이후 ∧ 기록한 논거와 무관, 세 조건을 코드로 고정한 결과입니다(초보자에게는 "자본을 줄이는 결정" 풀이도 함께).

### 정직 고지 — 한계를 숨기지 않는 것도 신뢰 설계의 일부

- **국내 상장주식 전용.** 해외주식은 아직 다루지 않습니다(확장 스텁도 없음).
- 게이트의 검출 범위는 "이 시스템의 산출물에서 권유 표현 0건 보장"이지 "세상 모든 권유 표현"이 아닙니다 — 범위를 부풀리지 않습니다.
- **Codex 실로드는 부분 확인입니다.** 그리고 이 부분 확인 방식 자체가 우리 철학의 일관성입니다. 검증 가능한 것과 규격이 필요한 것을 구분해서, **되는 건 실증하고 안 되는 건 정직하게 남겼습니다**:
  - **MCP 서버 실연결은 실증됨** — 로컬 Codex CLI(v0.143.0)에 서버를 등록하니, Codex 세션이 우리 도구 10종(dart_resolve_corp · dart_list_disclosures · dart_get_financials · dart_get_report_item · dart_get_events · dart_get_insider · price_get_daily · ledger_read · ledger_write · gate_check)을 전부 인식하고 요청에 응답했습니다. "코드가 Codex와 실연결된다"는 핵심이 증명됐습니다.
  - **플러그인 카탈로그 로드는 정적 검증으로 남김** — 이 버전 Codex는 플러그인을 마켓플레이스 스냅샷으로만 설치해서, 단일 플러그인인 우리에겐 별도 규격 작업이 필요합니다. 그래서 스킬 3종의 동작은 151개 테스트 + 컨트랙트 3종으로 증명했습니다.
  - 제품에서 "공시상 없으면 없다고 말한다"는 원칙을, 개발·검증 과정에서 "확인된 것과 못 한 것을 구분해 말한다"로 똑같이 지켰습니다.

> 【캡처 ③】 브리핑의 [원문] 링크로 열린 실제 DART 뷰어 화면
> 【캡처 ④】 로컬 Codex 세션에서 우리 도구 10종이 인식된 터미널(`codex mcp list` + 도구 목록 응답)

## 8. 로드맵 — 실무자의 Codex에서 A씨의 손끝까지

오늘은 실무자 검증 도구, 내일은 앱 속 '어땠지' 옆자리, 종착점은 센터장이 말한 그 에이전트입니다: *"굳이 앱이 나중에 있을까... 내가 어떤 의도를 가지고 액션을 했을 때 그걸 실행해 주는 애는 분명히 카카오페이증권의 에이전트일 거다."*

**입력 2개 → 0개.** 외부 도구인 오늘은 종목과 매수일 두 개를 입력받습니다. 그런데 카카오페이증권 안에서는 이 둘이 **이미 계좌 원장에 있습니다** — 사용자가 뭘 언제 샀는지 회사가 압니다. 입력이 0이 됩니다. 사용자는 아무것도 입력하지 않아도, 보유 종목을 여는 순간 "산 뒤로 뭐가 달라졌는지"가 떠 있습니다. **이 서비스를 증권사가 만들어야 하는 이유가 여기 있습니다.** (전환 설계: 케이스=실사용자 보유 종목, 원장에 user 차원 추가 — 부록 D 계좌 연동 행.)

그리고 그 전환은 껍데기만 바뀝니다. **돈이 많이 드는 교체(저장 DB·API 서버·계좌 연동)는 전부 껍데기이고, 이 제품의 본질 — 판정 룰·게이트·시점 필터·부재의 정직 — 은 한 줄도 다시 쓰지 않습니다.** 파이프라인을 CLI가 아니라 순수 함수로 유지한 것, 원장 스키마를 처음부터 DB급으로 정의한 것이 그 이유입니다.

## 9. 마무리

왜샀지는 답을 주는 AI가 아닙니다. 초보 투자자가 스스로 납득하게 만드는 AI입니다 — 카카오페이증권이 평가 기준으로 말한 '설득의 과정', 그 자체를 만들었습니다.
