# 03-DEV-PLAN — 왜샀지 (whybuy)

> 기반 문서: 01-SCENARIO(확정), 02-PRD(확정)
> 이 문서는 PRD를 "만드는 순서"로 번역한다. 개발은 게이트(G) 단위로 진행하며, 각 게이트는 **선행조건 → 작업 상세 → 산출물 → 완료 판정(실행 가능한 명령) → 시뮬레이션 검증 → 리스크와 대응**으로 정의된다. 완료 판정을 통과하지 못하면 다음 게이트로 넘어가지 않는다.

## 0. 개발 원칙과 환경

### 0.1 게이트 단위 순차 개발

한 게이트를 완성 → 테스트 → 기존 게이트와의 호환성 확인 → 커밋 → 다음 게이트. 게이트를 건너뛰거나 두 게이트를 한 커밋에 섞지 않는다. 각 게이트의 완료 판정은 사람의 인상이 아니라 **exit 0으로 끝나는 명령**이다.

### 0.2 결정 주권

다음 항목은 Claude가 발견·제안하고 **아엔이 승인해야 반영**된다. 자동 적용 금지:

- 픽스처 실데이터와 케이스 파라미터(`buy_date`, `as_of`) 확정 (G1 리허설 결과)
- case-003 배당주 최종 선정 (G1)
- 분류 라벨 세트 (G2 — Claude 초벌 라벨링, 아엔 검수 후 골든 세트 확정)
- 게이트 규칙 사전 `gate_rules.yaml`의 초안과 이후 모든 변경 (G4)
- 골든 파일(픽스처, 라벨 세트, 게이트 양성·음성 세트, 기대 브리핑 스냅샷)에 영향을 주는 모든 변경

### 0.3 두 개의 실행 환경

| 환경 | 네트워크 | 역할 |
|---|---|---|
| **아엔 로컬** | OpenDART·KRX 접근 가능 | G1 픽스처 수집, G9 링크 체커, (가능 시) Codex 라이브 확인 |
| **개발 환경** (Claude Code 등) | 외부 API 접근 불가 전제 | 그 외 전부 — 픽스처가 커밋된 뒤에는 완전 오프라인으로 개발·테스트·데모 재현 가능 |

이 분리가 성립하려면 **G1(픽스처)이 다른 모든 데이터 의존 게이트보다 앞서야** 한다. 단, G1이 아엔의 로컬 작업이므로 그동안 개발 환경이 놀지 않도록 병렬 트랙을 둔다 (0.5).

### 0.4 개발용 미니 픽스처 (dev-mini) — 실데이터 도착 전의 개발 차단 방지

- `data/fixtures/dev-mini/`: 실제 DART 응답 **형식을 그대로 본뜬 합성 데이터** (가상 회사 "테스트물산", 공시 제목은 실제 규격 패턴). 용도는 **단위 테스트와 파이프라인 배선 확인뿐**이다
- 규칙 두 개로 데모 오염을 차단한다: ①dev-mini 케이스는 `is_dev: true` 플래그를 가지며, `run_skill`은 이 플래그가 있는 케이스의 브리핑을 `reports/`에 저장하지 않는다 (tmp로만) ②제출물의 모든 `reports/` 산출물은 실존 케이스에서만 나온다 — "전 케이스 실존" 결정(PRD 9.3)과 충돌하지 않음
- 실데이터가 커밋되면 스냅샷 테스트(분류·판정·브리핑)는 전부 실데이터 기준으로 작성한다. dev-mini는 형식 검증용으로만 남는다

### 0.5 게이트 의존 그래프와 병렬 트랙

```
[개발 환경 트랙]                       [아엔 로컬 트랙]
G0 골격 ──────┐
              ├─ G2 분류기(dev-mini로 배선) ─┐
G4 게이트 ────┤                             ├─ G2' 분류 골든 확정(실데이터) ← G1
G3 원장+MCP ──┘                             │
                                            ▼
G1 픽스처 수집+리허설 (아엔 로컬) ──────→ G5 판정 엔진 → G6 → G7 → G8 → G9 통합
```

- G0, G3, G4는 실데이터가 필요 없다 → G1과 병렬 진행
- G2는 dev-mini로 코드를 먼저 만들고, G1 완료 후 실데이터 골든 세트로 확정(G2')
- G5부터는 실데이터 픽스처가 전제

### 0.6 키·로그 관리

- `DART_API_KEY`는 `.env` 파일로만 관리한다. 채팅·코드·커밋에 키를 노출하지 않는다 (`.env`는 gitignore, `.env.example`만 커밋)
- 모든 게이트의 완료 판정 명령 실행 로그를 `logs/gate-{N}-{date}.txt`로 보존한다 — 제출물의 "검증 증거" 부록이 된다

### 0.7 유지보수·확장 규약 (실무자 도구 → 고객 서비스 전환 대비)

이 플러그인은 오늘은 실무자 검증 도구지만 로드맵의 종착점은 고객 서비스다. 그 전환에서 **로직은 그대로 가고 껍데기만 바뀌도록**, 처음부터 네 가지 규약을 강제한다:

1. **설정 중앙화 (`config.yaml`)**: 급변 기준(±5%, z≥2), 시장 유사 밴드(±2%p), lookback(D-30~D+3), 약화 경계(50%), 보기 상한(8) 등 **모든 판단 파라미터를 한 파일에** 둔다. 코드 하드코딩 금지. 브리핑에 노출되는 경계값은 이 파일에서 읽어 표기한다 — 값과 표기가 어긋나는 사고를 구조적으로 차단
2. **문구 단일 출처 (`tools/textstore.py`)**: 면책 문구, 기준일 문구, "반증은 회사 평가가 아니라..." 정의 문구, 거울 고정 질문 — **게이트가 검사하는 문구와 템플릿이 출력하는 문구는 반드시 같은 상수를 임포트**한다. 두 곳에 각각 적으면 한쪽만 고쳤을 때 정상 브리핑이 게이트에서 전멸하는 사고가 난다 (문구 수정 = 상수 1곳 수정으로 끝나야 함)
3. **원장 접근 격리 (`mcp/ledger_store.py`)**: 원장 읽기·쓰기는 이 모듈만 거친다. 오늘은 JSON 파일, 고객 서비스에서는 DB — **이 모듈 하나만 교체하면 되는 구조**. 스킬·판정 엔진 코드는 저장 방식을 모른다
4. **논거 유형 추가 체크리스트**: 새 유형 추가 시 반드시 함께 변경해야 하는 5곳을 문서로 고정 — ①PRD 5.3 카탈로그 표 ②verdict_engine 룰 ③test_verdict_engine 케이스 ④라이브러리 노출 규칙 ⑤원장 스키마 enum. 룰-카탈로그 일치 테스트가 누락을 기계적으로 잡는다

### 0.7 확장성·유지보수 설계 원칙 (고객 서비스 전환 전제)

이 플러그인의 오늘 사용자는 실무자지만, 종착점은 고객 서비스다 (시나리오 2번 섹션 로드맵). 따라서 "나중에 갈아끼울 것"들을 지금 한 곳씩에 모아두는 다섯 원칙을 전 게이트에 적용한다:

1. **설정의 단일화** — 모든 경계값(약화 판정 50%, 급변 기준 5%·z≥2, 시장 유사 ±2%p, lookback D-30~D+3, 보기 상한 8, 급변 표기 상한)을 `config/whybuy.yaml` 한 파일에 모은다. 코드에 숫자를 흩뿌리지 않는다. 브리핑에 경계값을 노출한다는 원칙은 유지하되, 노출되는 값도 이 파일에서 읽는다 — 서비스팀이 코드 수정 없이 정책을 조정할 수 있는 구조
2. **필수 문구의 단일 출처** — 면책 문구, 기준일 문구, 판정 정의 문구, 거울 고정 질문은 `config/phrases.yaml`에 한 번만 정의하고, **템플릿과 컴플라이언스 게이트가 같은 파일을 읽는다.** 문구가 두 곳에 따로 정의되면 한쪽만 수정됐을 때 게이트가 정상 브리핑을 차단하는(또는 문구 누락을 놓치는) 표류 사고가 난다 — 이 표류가 이번 검토에서 찾은 가장 위험한 유지보수 함정이다
3. **템플릿의 외재화** — 브리핑 템플릿을 코드 문자열이 아니라 `templates/*.md` 파일로 둔다. 고객 서비스 전환 시 문안 수정은 서비스팀 일이지 엔지니어 일이 아니어야 한다. 게이트가 최종 방어선이므로 템플릿 수정이 규제 선을 넘으면 기계가 잡는다
4. **원장 접근의 단일 진입점** — `reasons.json` 읽기·쓰기는 `tools/ledger_store.py` 모듈만 한다 (스키마 검증 내장). 고객 서비스 전환 시 "파일 → DB" 교체가 이 한 모듈의 교체로 끝난다. `schema_version` 필드가 마이그레이션의 앵커
5. **룰 레지스트리 패턴** — 논거 유형 추가 = ①카탈로그 표에 행 추가 ②룰 함수 등록 ③테스트 케이스 추가, 이 3단계로 끝나야 한다. 기존 룰 코드를 건드리지 않고 확장되는지가 G5 리뷰 기준

부록 D에 이 원칙들이 실제 서비스 전환 때 어떻게 이어지는지의 체크리스트를 둔다.

## G1 — 픽스처 수집 + 케이스 리허설 (아엔 로컬, 이 계획의 최대 불확실성 소거 지점)

**선행조건**: G0. 아엔의 OpenDART 인증키 발급 (무료, 즉시 발급) 및 `.env` 세팅

**작업 상세**

1. `tools/fetch_dart.py` 작성 (Claude가 개발 환경에서 작성 → 아엔이 로컬 실행):
   - corpCode.xml zip 다운로드 → 삼성전자(005930), 카카오(035720), 배당 후보 3종(KB금융 105560, SK텔레콤 017670, KT&G 033780)의 corp_code 추출 → `fixtures/corp/`
   - **종목별 기업개황 API(apiId 2019002) 1회 호출 → 시장 구분을 `fixtures/corp`에 저장.** DART가 주는 원래 코드 `corp_cls`(Y/K/N/E)는 API 응답 쪽이라 우리가 못 바꾸지만, 픽스처에 뭐라고 저장할지는 우리 몫이다 — 그러니 `fetch_dart.py`가 받는 즉시 읽기 좋은 값으로 번역해 저장한다: **Y → `kospi`, K → `kosdaq`, N → `konex`, E → `etc`**. corpCode.xml에는 시장 구분이 없으므로(corp_code·회사명·종목코드·수정일만 포함) 이 호출이 없으면 시장 대비 태그가 종목별 올바른 지수(코스피/코스닥)를 선택할 근거가 없다. 종목당 호출 1회 추가라 한도 비용은 사실상 없음
   - 종목별 공시 목록 수집: 유형 **A(정기)/B(주요사항)/C(발행)/D(지분)/E(기타)/I(거래소)** 6종을 **각각 별도 호출**로 (1단 분류가 API 수준에서 확정되도록). **E와 I가 필수인 이유 (API 명세 대조에서 확인)**: 자기주식취득/처분은 E001(기타공시), 잠정실적은 I002(공정공시), 단일판매·공급계약과 신규시설투자 등은 I001(수시공시)로 제출된다 — A~D만 수집하면 reason-recall의 핵심 후보(잠정실적)와 화이트리스트 이벤트 다수가 통째로 실종된다. 기간은 각 케이스 후보 매수일 기준 D-60 ~ 오늘, 제출일 포함 → `fixtures/disclosures/{corp_code}.json`
   - **페이지네이션 필수**: `page_count` 최대 100 — 대형주는 100건을 초과하므로 `total_page`를 읽어 전 페이지 순회. `last_reprt_at=N`(기본, 정정 포함 전체 수집)으로 받고, 응답의 `rm` 필드('정'=정정신고 존재)와 `[기재정정]` 접두를 그대로 저장 (정정 이력 표시의 원천)
   - 분기 재무 주요 계정: 2025 Q1 ~ 최신 제출분까지 (YoY 비교에 전년 동기가 필요하므로 **판정 대상 분기보다 1년 이전 분기까지** 수집 — 이게 빠지면 RULE-EARN-01이 계산 불능) → `fixtures/financials/`
   - 배당·최대주주·감사의견·자기주식 현황 항목: DS002 계열 API (호출 규격 확인됨 — `corp_code` + `bsns_year` + `reprt_code`(1분기 11013 / 반기 11012 / 3분기 11014 / 사업 11011), 2015년 이후만 제공). **배당 API는 당기·전기·전전기 3개년을 한 응답에 병렬 제공**하므로 RULE-DIV의 비교값이 호출 1회로 확보되고, 응답에 rcept_no가 포함되어 원문 링크 원칙도 성립 → `fixtures/report_items/`
   - 일별 시세 + 지수(코스피·코스닥): pykrx로 수집. **시세와 지수는 매수일 기준 최소 30거래일 이전부터 수집한다 (20거래일 z-score 롤링 창 + 안전 마진 10거래일).** 필요한 롤링 창은 정확히 20거래일이지만, 마진 5일은 휴장일 계산 실수·데이터 결측 같은 자잘한 변수에 취약하다 — 시세 며칠 더 받는 건 비용이 없으므로 마진 10거래일로 잡는다. "같은 기간"으로 읽어 매수일부터만 수집하면 매수일 직후 구간의 z-score가 에러 없이 조용히 틀린 값을 내므로, 애매한 표현을 산술 조건으로 못박는 수정이다 → `fixtures/prices/{ticker}.csv`, `fixtures/prices/index_{kospi|kosdaq}.csv`
   - 수집 로그와 호출 수를 출력 (일일 한도 대비 사용량 가시화)
   - **재실행 안전(idempotent)**: 이미 존재하는 픽스처 파일은 스킵하고 `--force`로만 갱신. API 오류는 지수 백오프로 3회 재시도, 부분 실패 시 실패 목록을 출력하고 exit 1 — 아엔이 "다시 실행"만 하면 이어서 받는 구조 (네트워크 불안정이 수집 전체를 무효화하지 않게)
2. **케이스 리허설** — 수집된 실데이터 위에서:
   - **유형 코드 매핑 검증 (독립 체크포인트 — 분류기 코드(G2) 작성 전 선행)**: 화이트리스트 이벤트(실적, 배당, 자기주식, 증자와 CB, 최대주주 변동, 감사의견, 소송과 제재, 시설투자와 공급계약, 사업 중단)가 실제로 어느 유형 코드(A부터 E, I)로 들어오는지 실데이터로 대조하고, 기대와 다른 항목은 REHEARSAL.md에 기록한다. 기대한 공시가 예상한 유형 서랍에 없으면(예: 어떤 이벤트가 B가 아니라 I로 들어오면) 분류기를 짜기 전에 알아야 수집 스크립트(작업 1) 단에서 고칠 수 있다 — 그래서 G2 첫 작업이 아니라 G1 완료 판정에 붙는 독립 체크포인트로 올린다
   - case-001(삼성전자): "유효 판정 + 하락 구간 + 나빠진 공시 0건" 장면이 성립하는 `buy_date`/`as_of` 조합 탐색
   - case-002(카카오): 급변일(±5% 또는 z≥2)이 존재하고, [긍정 재료 → 이후 약화/반증]의 논거 궤적이 성립하는 조합 탐색. 불성립 시 PRD 제약 8의 폴백 사다리 적용
   - case-003: 배당 후보 3종의 배당 공시 이력을 대조해 "유효/약화 장면이 가장 선명한 1종"을 Claude가 제안
   - 결과를 `docs/REHEARSAL.md`로 작성: 각 케이스의 확정 파라미터, 장면 성립 근거(해당 공시 rcp_no 목록), 폴백 사용 여부
3. **아엔 승인**: REHEARSAL.md 검토 → 케이스 3종 파라미터 확정 → `fixtures/cases.json` 커밋

**산출물**: 픽스처 전체 + REHEARSAL.md + cases.json

**완료 판정**
```

python tools/rehearsal_check.py                 # cases.json의 각 케이스에 대해: 매수일 이후 공시 ≥1건,

```
+ 아엔 승인 기록 (REHEARSAL.md에 서명 줄)

**시뮬레이션 검증** (case-001을 끝까지 굴려본 결과):
- 수집: corpCode 매핑 → 유형별 목록 호출 → 재무 API — 전부 문서화된 표준 엔드포인트, 호출량은 종목당 수십 회 수준으로 일일 한도 대비 여유

- **여기서 발견한 필수 조건 하나**: YoY 비교를 위해 전년 동기 재무까지 수집해야 한다는 것 (작업 상세에 반영). 이걸 빠뜨리면 G5에서 되돌아와야 함

## G2 — 공시 분류기 (dev-mini로 배선 → 실데이터로 골든 확정)

**선행조건**: G0 (배선), G1 (골든 확정)

**작업 상세**
1. `tools/classify.py`: 2단 분류 — 1단은 픽스처의 유형 필드(A/B/C/D, G1에서 호출 단위로 확정됨) 그대로 사용, 2단은 제목 키워드 사전(`tools/classify_rules.yaml`)으로 세부 유형 매핑. **매칭 전 제목 정규화**(공백 제거, 전각·반각 괄호 통일, `ㆍ`류 특수문자 통일) — DART 제목은 규격적이지만 공백·괄호 변형이 존재하므로 정규화 없이는 사전이 흔들린다. `[기재정정]` 접두는 원 유형으로 분류하되 `is_correction: true` 플래그
2. 키워드 사전 초안: PRD 5.3 카탈로그의 유형 + 화이트리스트 유형(실적/배당/자기주식/증자·CB/최대주주/감사의견/소송·제재/시설투자·공급계약/사업중단). 미매칭은 `etc`
3. dev-mini로 단위 테스트 → G1 완료 후: 실데이터 공시 목록 전건에 대해 Claude가 초벌 라벨링 → **아엔 검수** → `tests/labels/{corp_code}.json` 골든 세트 확정 → `tests/test_classify.py` 스냅샷 테스트
4. 골든 확정 후 사전 수정이 생기면 라벨 세트와 함께만 변경 (결정 주권 항목)

**산출물**: classify.py, classify_rules.yaml, 골든 라벨 세트, test_classify.py

**완료 판정**
```

python tools/classify.py --stats fixtures/disclosures/  # 유형별 분포 출력 (etc 비율 가시화)
```

**시뮬레이션 검증**: 삼성전자·카카오의 실제 공시 제목은 "분기보고서 (2026.03)", "주요사항보고서(전환사채권발행결정)", "임원ㆍ주요주주특정증권등소유상황보고서"처럼 규격화되어 있어 키워드 매핑의 정확도가 구조적으로 높다. 관건은 **양** — 대상 기간(케이스당 4~8개월)의 두 대형주 공시는 지분공시 포함 수백 건에 달할 수 있다. 대응: 라벨링 범위를 "2단 분류가 화이트리스트로 매핑한 것 전건 + etc 중 무작위 표본 30건"으로 정의한다. etc 전건 라벨링은 불필요하다 — etc는 본문 미표기·접힘 대상이라 오분류의 피해가 "중요 공시를 etc로 잘못 접는 것"에 집중되고, 그건 화이트리스트 매핑 검수와 etc 표본 검사로 잡힌다. (이 범위 정의 자체를 골든 세트 문서에 명시해 검증 방법론의 정직성을 유지)

**리스크**: 키워드 사전의 과소 매칭(중요 공시가 etc로) — 위 표본 검사로 검출. 과대 매칭(잡음이 화이트리스트로) — 골든 세트가 검출.

## G3 — 원장 스키마 + MCP 서버 (픽스처 모드)

**선행조건**: G0 (G1과 병렬 가능 — dev-mini로 개발)

**작업 상세**
1. `data/ledger/reasons.json` 스키마 정의 (PRD 5.1) + `tools/validate_ledger.py` (jsonschema 기반: 필수 필드, 날짜 형식, rcp_no 14자리, status·type·source enum) + `mcp/ledger_store.py` (0.7-3의 격리 모듈 — 읽기·쓰기·스키마 검증을 이 한 곳에서)
2. `mcp/dart_client.py`: 픽스처/캐시 읽기 계층. `WHYBUY_MODE=fixture`(기본)면 fixtures만, `live`면 API 호출+캐시 저장. **as_of 필터를 이 계층에서 일괄 적용** (모든 도구가 공짜로 시점 시뮬레이션을 얻도록)
   - **DART 응답 status 코드 구분 처리 (명세 확인됨)**: `000`=정상 / **`013`=조회된 데이터 없음 → 예외가 아니라 "부재" 상태로 반환** (배당 무이력 종목 조회 등은 정상적인 부재이며, 이걸 예외로 던지면 부재의 정직 파이프라인이 깨진다) / `020`=일일 한도(20,000건) 초과 → 라이브 모드 중단·캐시 보존 후 명확한 안내 / `010`·`011`=키 문제 → 설정 안내. 상태 코드→처리 매핑을 dart_client 한 곳에 표로 구현
3. `mcp/server.py`: PRD 6번의 도구 10종을 stdio MCP로 노출. 각 도구는 dart_client의 얇은 래퍼 + 입력 검증. **도구는 실패를 예외가 아니라 구조화된 오류 응답(`{error, message, hint}`)으로 반환** — 존재하지 않는 case·ticker, 픽스처 범위 밖 as_of, 데이터 부재 등. 예외가 그대로 터지면 에이전트가 상황을 읽고 복구할 수 없다
4. 도구 단위 테스트: dev-mini 대상, as_of 필터 동작(기준일 이후 제출분 미노출) 포함

**산출물**: 스키마, validate_ledger.py, dart_client.py, server.py, 도구 테스트

**완료 판정**
```
pytest tests/test_mcp_tools.py -q               # 도구 10종 단위 + as_of 필터
python tools/validate_ledger.py data/ledger/reasons.json
timeout 5 python mcp/server.py < tests/mcp_smoke_input.jsonl   # stdio 기동 + initialize/list_tools 응답 스모크
```

**시뮬레이션 검증**: stdio MCP 서버는 표준 패턴이고, 도구가 전부 "파일 읽고 필터링"이라 복잡도가 낮다. 시뮬레이션에서 확인한 설계 포인트 하나 — **as_of를 각 도구가 아니라 dart_client 한 곳에서 강제**해야 도구 추가 시 시점 누출 버그가 구조적으로 불가능해진다 (작업 2에 반영). 스모크 테스트의 입력(jsonl)은 MCP initialize 핸드셰이크를 흉내 내는 고정 파일로, Codex 없이도 서버 생존을 판정한다.

**리스크**: MCP 파이썬 SDK 버전에 따른 인터페이스 차이 → requirements에 버전 고정.

## G4 — 컴플라이언스 게이트 (실데이터 불필요 — G1과 병렬)

**선행조건**: G0

**작업 상세**
1. `tools/gate_rules.yaml` 초안: 금지 패턴(지시형·권고형·예측형 종결 표현 — "~하세요", "~하시기 바랍니다", "~것으로 예상/전망", "저평가/고평가/유망/기회", "확실히/분명히 + 상승·하락"), 필수 요소(기준일 문구, 면책 문구, 판정 라인당 링크, 판정 브리핑의 정의 문구), 허용 예외(거울 모드 고정 질문 문구) — **초안을 아엔에게 제출·승인 후 확정** (결정 주권)
   - **판정 상태별 필수 요소 차등 (검토에서 발견된 정합 결함의 수정)**: 유효/약화/반증 판정 라인은 원문 URL 필수. 그러나 **확인불가·데이터 부재 판정은 인용할 원문 자체가 존재하지 않으므로** URL을 강제하면 "부재의 정직" 원칙과 게이트가 서로 충돌한다 (부재를 정직하게 말하는 브리핑이 게이트에서 차단되는 모순). 이 상태들의 필수 요소는 URL 대신 "대조를 시도한 데이터 소스와 부재 사실" 문구로 정의한다. 게이트는 판정 라인의 상태를 읽어 상태별 규칙을 적용한다
2. `tools/compliance_gate.py`: yaml 로드 → 마크다운 검사 → exit code + 위반 리포트 + `reports/{case}/gate-log-*.txt`
3. `tests/gate/positive/` (통과해야 하는 샘플 — "임원이 주식을 매도했다는 공시", "매수 논거", "+32%", "시장 대비 하회" 등 사실 서술 포함) / `tests/gate/negative/` (차단해야 하는 샘플 — 전 금지 카테고리 최소 2문장씩, 우회 변형 포함: "파는 게 낫습니다", "지금이 저가 매수 기회")
4. `tests/test_gate.py`: 음성 검출률 100% + 양성 오탐 0

**산출물**: gate_rules.yaml(승인본), compliance_gate.py, 양성·음성 세트, test_gate.py

```
 → G5 verdict(reason+financials)     [입력 충족: G1에서 전년 동기까지 수집 — 본문에서 선제 반영]

 → G7 recall(후보→원장)              [입력 충족: 결정성 규칙으로 후보 ID 고정]
 → G8 audit(원장+판정+조립)           [입력 충족]
 → G9 통합                           [전 산출물 존재]
```

**끊긴 고리 없음.** 관통 시뮬레이션에서 발견되어 본문에 선제 반영된 것 4건: ①전년 동기 재무 수집 필수(G1) ②적자↔흑자 전환의 YoY 부호 분기(G5) ③정적 서술 경로 필수(G6) ④신규 중대 사실의 3조건 정의(G8).

## 부록 B — 남는 한계 (정직 고지)

1. **Codex 실로드 검증**은 개발 환경에서 불가능 — 아엔 로컬/협력 계정 시도, 최종 폴백은 정적 검증+고지 (G9)
2. **게이트의 검출 범위**는 "이 시스템의 산출물"이지 "세상 모든 권유 표현"이 아님 — 범위를 명시하는 것으로 방어 (G4)
3. **정정공시 계보 추적**은 휴리스틱 (PRD 제약 4)
4. **대화형 UX의 품질**(질문을 자연스럽게 내미는가)은 자동 검증 불가 — SKILL.md 지시문 리뷰로만 관리 (G7)
5. **수기 라벨링의 커버리지**는 "화이트리스트 전건 + etc 표본"으로 정의된 범위 안에서만 보증 (G2)

## 부록 D — 고객 서비스 전환 경로 (확장성 명세)

로드맵의 종착점(앱 내 '왜샀지' → 에이전트)으로 갈 때 **무엇이 그대로 가고 무엇이 교체되는지**를 지금 명시해 둔다. 0.7의 규약들이 이 표를 성립시키는 장치다.

| 구성 요소 | 실무자 도구 (지금) | 고객 서비스 (전환 시) | 전환 비용 |
|---|---|---|---|
| 판정 룰 엔진 | verdict_engine.py | **그대로** — 순수 함수라 어디서든 호출 | 0 |
| 컴플라이언스 게이트 | 출력 전 기계 검사 | **그대로** — 오히려 고객 노출에서 가치가 커짐 | 0 |
| 분류기 + 사전 | classify.py + yaml | **그대로** — 사전 갱신 절차만 운영 프로세스화 | 낮음 |

| 문구·설정 | textstore / config.yaml | **그대로** — 문구 심의(법무 검토)가 상수 파일 리뷰로 끝남 | 낮음 |
