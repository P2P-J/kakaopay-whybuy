# 02-PRD — 왜샀지 (whybuy)

> 기반 문서: 01-SCENARIO-waesatji.md (확정본)
> 이 문서는 시나리오의 장면들을 "만들 수 있는 것"으로 번역한다. 모든 기능은 **상황 → 문제 → 접근 → 해결**의 구조로 서술하고, 설계 과정의 고민은 9번 섹션(설계 결정 기록)에 트레이드오프와 함께 남긴다.

---

## 1. 제품 정의

### 1.1 한 줄 정의

초보 투자자의 매수일을 기준으로 공시 사실을 재정렬하고, 산 이유를 객관식으로 찾아주며, 그 이유가 아직 유효한지 공시 원문으로 판정하는 Codex 플러그인.

**추천하지 않는다. 판정한다.** 종목을 골라주지도, 매매 시점을 알려주지도 않는다 — 사용자 자신의 판단 근거를 공식 기록에 대조해주는 사실 감사(audit) 도구이며, 출력의 종착점은 언제나 "판단은 당신 몫이고, 근거는 여기 있습니다"이다.

### 1.2 이름

- **한국어**: 왜샀지 — 초보 투자자가 가장 많이 하는 혼잣말 "이 주식 왜 샀지?"에서 물음표를 뗀 것 (아엔 명명. 카카오페이증권의 기존 서비스 '어땠지'를 모른 채 독립적으로 도출된 이름이며, 사후에 제품군 정합이 발견됨)
- **영어·코드명**: `whybuy` (아엔 확정)
- 포지셔닝: 카카오페이증권의 '어땠지'가 **시장의 시간**("오늘 시장 어땠지?")에 답한다면, 왜샀지는 **나의 시간**("내가 산 뒤로 뭐가 달라졌지?")에 답한다

### 1.3 배경 — 왜 이 제품인가 (요약)

상세 논증은 시나리오 문서(01-SCENARIO) 1·4·6번 섹션에 있고, 여기서는 설계의 전제가 되는 세 가지만 요약한다.

1. **초보 투자자의 손실 메커니즘**: 국내 개인투자자 실증 연구에 따르면 이익 종목을 매도할 확률이 손실 종목의 2배 이상(처분효과)이고, 이 경향은 투자경험이 부족할수록 강하며 성과를 해친다. 뿌리는 매도 판단의 유일한 기준이 "내 매수가 대비 손익률"이라는 것 — 매수할 때 근거가 없었으니 팔 때 판단할 기준도 가격뿐이다
2. **기존 서비스의 구조적 공백**: 뉴스(오늘), 변동 원인 분석(가격이 움직인 순간), 어닝콜(실적 일정) — 모든 정보 서비스가 세상의 달력 위에 서 있다. "내가 산 뒤로 뭐가 달라졌어?"라는, 초보자가 실제로 하는 질문에 답하는 서비스는 국내외에 없다 (경쟁 조사 완료)
3. **규제 제약이 만드는 설계 공간**: 개인화된 매수·매도 지시는 투자자문·투자권유 규제의 영역이라 증권사가 출시할 수 없다. 왜샀지는 규제를 피하는 게 아니라 처음부터 선 안쪽에서 설계됐다 — 추천이 아니라 사용자 논거에 대한 사실 감사이므로, 규제가 금지하는 행위 자체가 제품에 존재하지 않는다

### 1.4 제품 형태와 구성

Codex 플러그인 1개. 내부 구성은 네 덩어리다:

| 구성 | 역할 | 답하는 질문 |
|---|---|---|
| 스킬 `buy-timeline` | 매수일 앵커 타임라인 + 변동–공시 매칭 + 수익률·시장 대비 표기 | "내가 산 뒤로 뭐가 달라졌어?" |
| 스킬 `reason-recall` | 질문 세트(객관식+직접 입력)로 매수 이유를 재인 방식으로 수집 | "나 이거 왜 샀더라?" |
| 스킬 `thesis-audit` | 기록된 논거를 최신 공시에 대조해 유효/약화/반증/확인불가 판정 (+거울 모드) | "그때 그 이유, 아직 유효해?" |
| 공통 기반 | 로컬 MCP 서버(DART 조회), 논거 원장(`reasons.json`), 컴플라이언스 게이트, 픽스처 데이터 | — |

### 1.5 사용자와 수혜자

- **주 사용자 — 카카오페이증권 AI서비스센터 실무자**: 이 플러그인은 "왜샀지를 서비스로 출시해도 되는가"를 실험하는 검증 도구다. 실무자의 워크플로: ①초보 투자자 페르소나 케이스(종목·매수일)를 로드하고 ②스킬을 실행해 브리핑을 산출한 뒤 ③게이트 통과 로그, 원문 링크, 판정 rule_id를 근거로 품질·규제 준수를 검증한다. 판정 하나하나를 공시 원문과 대조해 재현할 수 있다는 것이 이 도구의 핵심 가치다
- **최종 수혜자 — 초보 투자자 (페르소나 A씨)**: 모든 브리핑의 수신자. A씨의 여정이 곧 스킬의 순서다: 불안(-12%) → 타임라인(사실을 본다) → 재인(이유가 생긴다) → 판정(이유의 생사를 확인한다) → 거울(가격인가, 사실인가). 브리핑의 문체 기준: 금융 초보가 읽는 언어로 쓰고, 불가피한 용어(전환사채 등)는 괄호로 풀이한다

### 1.6 목표 (이번 제출물에서 달성하는 것)

1. 스킬 3종(`buy-timeline`, `reason-recall`, `thesis-audit`)이 픽스처 케이스에 대해 완전한 브리핑을 산출한다
2. 모든 브리핑이 컴플라이언스 게이트를 기계 검증으로 통과한다 (권유성 표현 0건, 전 판정에 원문 링크, 면책 문구 존재)
3. 실무자가 판정을 재현·검증할 수 있다 (같은 입력 → 같은 판정, 판정마다 대조 규칙 ID와 공시 원문 명시)
4. Codex 라이브 실행이 불가한 환경에서도 정적 검증(컨트랙트 명령어 exit=0, MCP stdio 기동, 산출물 동봉)으로 품질을 증명한다

네 목표 모두 사람의 인상이 아니라 **기계 검증으로 판정 가능**하도록 정의했다 — 각 목표는 11번 검증 계획의 항목과 1:1로 대응한다.

### 1.7 비목표 (명시적으로 하지 않는 것)

- 매매 실행, 주문 연동, 실계좌 연동 — 하지 않는다
- 시세 예측, 목표가, 가치 평가 — 규제상·철학상 하지 않는다
- 뉴스·커뮤니티·리서치 수집과 분석 — 판정 근거로 쓰지 않는다 (원칙 1·3)
- 해외주식 — **아직** 다루지 않는다 (아엔 확정). 이번 제출물에는 확장 스텁·어댑터 인터페이스도 포함하지 않으며, README에 "국내 전용, 해외는 아직 다루지 않음"을 명시한다
- 실시간성 — 속보 서비스가 아니다. 브리핑은 항상 데이터 기준일을 명시한 시점 스냅샷이다

### 1.8 규제 스탠스 (요약)

출력에 존재할 수 있는 것은 두 종류뿐이다: **공시 원문의 인용**과 **산술 비교의 결과**. 종목 추천, 매매 지시, 미래 예측, 가치 평가는 존재하지 않는다. 이 스탠스는 시나리오 문서 4번 섹션의 4원칙(공식 기록 원문만 인용 / 해석·권유 금지 / 부재의 정직 / 판정 근거 표시)으로 정의되고, 본 문서 7번 섹션의 컴플라이언스 게이트로 기계 강제된다.

### 3.2 역할 분리 원칙 (하이브리드 설계)

**상황**: LLM에게 판정까지 맡기면 개발이 빠르지만, 같은 입력에 다른 판정이 나올 수 있고(비재현성), 환각 위험이 판정의 신뢰를 무너뜨린다.

**결정**: 결정적이어야 하는 것은 스크립트, 언어적이어야 하는 것은 에이전트.

- **스크립트(결정적)**: 공시 수집·필터링, 재무 수치 비교, 급변일 감지, 판정(룰 엔진), 게이트 검사
- **에이전트(언어적)**: 공시 제목·본문에서 이벤트 유형 분류 보조, 판정 결과를 초보자 언어의 브리핑 문장으로 서술
- **안전핀**: 에이전트가 쓴 모든 문장은 게이트를 통과해야만 파일로 확정된다. 에이전트의 서술 자유도가 규제 선을 넘을 수 없는 구조

### 3.3 시점 시뮬레이션 (`as_of`) — 데모의 핵심 장치

**상황**: 시나리오의 장면 3은 "매수 후 시간이 흐른 뒤"를 요구한다 — 경과 기간은 임의다 (1주일 뒤든 1년 뒤든 동작이 같아야 하고, 짧은 경과에서는 "변한 사실 없음"이 정상 출력이다). 데모에서 실제로 시간을 기다릴 수는 없다.

**해결**: 모든 MCP 도구와 스킬이 `as_of`(데이터 기준일) 파라미터를 받는다. `as_of=2026-04-15`로 실행하면 그 날짜까지 제출된 공시만 보이고, `as_of=2026-06-10`으로 다시 실행하면 그 사이의 공시가 추가로 보인다. 픽스처는 시점별 스냅샷이 아니라 **전체 공시 목록 + 제출일 필터**로 구현한다 (스냅샷 중복 없이 시간 여행 가능). 모든 브리핑 상단의 "이 브리핑은 YYYY-MM-DD까지 제출된 공시 기준입니다" 문구가 이 파라미터에서 나온다.

## 4. 기능 명세 — 스킬 3종

### 4.1 `buy-timeline` — 매수일 앵커 타임라인

**입력**
- `case`: 케이스 ID (픽스처) 또는 `{ticker, buy_date}` 직접 입력
- `as_of`: 데이터 기준일 (기본: 오늘)
- `with_price_overlay`: 변동–공시 매칭 포함 여부 (기본: true)

**처리 파이프라인**
1. 종목코드 → DART corp_code 변환 (MCP: `dart_resolve_corp`)
2. `buy_date` ~ `as_of` 구간의 공시 목록 수집 (MCP: `dart_list_disclosures`) — 정기공시(A)·주요사항보고(B)·발행공시(C)·지분공시(D)·기타공시(E)·거래소공시(I) 6유형. E와 I가 필수인 이유: 자기주식취득/처분(E001), 잠정실적 공정공시(I002), 단일판매·공급계약과 신규시설투자 등 수시공시(I001)가 여기에 제출된다
3. 이벤트 분류: 공시 제목을 이벤트 유형 카탈로그(5.3)에 매핑. 스크립트의 키워드 매핑이 1차, 미분류 항목만 에이전트가 분류 보조 (분류 결과는 원문 제목과 함께 표시)
4. **노이즈 필터**: 대형주는 구간 내 공시가 수십~수백 건에 달한다 (임원 소액 지분변동, 기업설명회 개최 등 반복성 공시 다수). 타임라인 본문에는 **화이트리스트 유형만 표기**하고 (실적, 배당, 자기주식, 증자·CB, 최대주주 변동, 감사의견, 소송·제재, 시설투자·공급계약, 사업 중단), 나머지는 말미에 "그 외 공시 N건 (전체 목록 링크)"로 접는다. 무엇을 숨겼는지도 숨기지 않는다
   - **분류의 정확도가 이 스킬의 품질을 결정하므로 2단 체계로 설계한다**: 1단 = DART 공시 유형 코드(정기 A / 주요사항 B / 발행 C / 지분 D / 기타 E / 거래소 I)로 대분류, 2단 = 공시 제목 키워드 사전으로 세부 유형 매핑. 어느 단에서도 확정 못 한 공시는 버리지 않고 "기타"로 접는다 (누락 제로 원칙)
   - **분류 테스트 세트**: 실존 픽스처 종목(삼성전자·카카오 등)의 실제 공시 목록 전체에 대해 기대 분류를 수기 라벨링한 스냅샷 테스트를 만든다 (`tests/test_classify.py`). 픽스처 수집 직후 최우선으로 수행 — 분류가 틀리면 타임라인·후보 생성·판정이 전부 오염되기 때문
5. "변하지 않은 것" 산출: 감시 대상 고정 항목(최대주주, 감사의견, 배당 정책)을 매수일 직전 상태와 `as_of` 상태로 비교 (MCP: `dart_get_report_item`). **감사의견은 연 1회(사업보고서) 항목이므로 "최근 사업보고서 기준"을 값과 함께 표기**
6. `with_price_overlay=true`이면: 일별 시세에서 급변일 감지(`detect_moves.py`, 기준: 일간 변동률 절대값 ≥ 5% 또는 20거래일 롤링 z-score ≥ 2) → 각 급변일의 [D-1, D+1] 창에서 공시 대조 → 매칭 또는 "공식 기록상 없음" 표기
7. **보유 수익률 산출**: 기준가 대비 `as_of` 종가 수익률 계산 + 같은 기간 시장 지수(종목 소속 시장에 따라 코스피 또는 코스닥) 수익률 병기 → 산술 비교 태그 부여: **시장 대비 상회 / 유사(±2%p 이내) / 하회**. "좋음/나쁨" 같은 평가어는 쓰지 않는다 — 태그는 계산 결과이지 판단이 아니다
   - **기준가 규칙**: 케이스에 `buy_price`(실제 체결가)가 있으면 그것을, 없으면 매수일 종가를 쓴다. 종가 기준일 때는 "실제 체결가와 다를 수 있는 참고치"임을 브리핑에 명시한다 — 사용자의 앱 화면 수익률과 숫자가 다르면 신뢰가 깨지므로, 다를 수 있는 이유를 먼저 말한다
   - **휴장일 규칙**: 매수일 또는 `as_of`가 휴장일이면 직전 거래일 종가를 쓰고 그 날짜를 표기한다
   - 배당 미반영 명시 (기존 규칙 유지)
8. 브리핑 마크다운 생성 (에이전트 서술) → 컴플라이언스 게이트 → `reports/`에 저장

**출력**: `reports/{case_id}/timeline-{as_of}.md`

```markdown
# {buy_date}, 당신이 산 이후 {회사명}에 일어난 일
> 이 브리핑은 {as_of}까지 제출된 공시 기준입니다. (매수 후 {D}일 경과)
> 보유 수익률(참고): {±N%} (기준: {실제 체결가 | 매수일 종가 — 실제 체결가와 다를 수 있는 참고치}, 배당 미반영) · 같은 기간 {코스피/코스닥} {±M%} → 시장 대비 {상회/유사/하회}

## 변한 것
- {날짜}: {이벤트 요약 — 사실 서술} [{공시명}, 원문]({dart_url})
...

## 변하지 않은 것
- 최대주주: {값} (매수 시점과 동일) [근거 공시 원문]({url})
...

## 주가가 크게 움직인 날 (참고)
- {날짜}: {±N%} ← {공시 매칭 결과 또는 "해당 기간 공시 없음. 회사의 공식 기록상 변한 사실은 없습니다 (시장 전체 흐름이나 수급 요인일 수 있으며, 이는 공시로 확인되지 않는 영역입니다)"}

---
본 브리핑은 공개된 공시 사실의 정리이며, 특정 종목의 매매를 권유하는 투자자문이 아닙니다.
```

**수용 기준 (acceptance criteria)**
- 매수일 이전 공시가 타임라인에 0건 포함
- "변한 것"의 모든 항목에 rcp_no 기반 원문 URL 존재
- 급변일 매칭에서 공시가 없는 경우 "없음" 문구가 정확히 출력 (설명 생성 금지)
- 게이트 exit 0

### 4.2 `reason-recall` — 역방향 논거 재구성 (객관식)

**입력**
- `case`: 케이스 ID
- `lookback`: 후보 수집 창 (기본: 매수일 기준 D-30 ~ D+3)
- `as_of`: 후보 생성 시점 (기본: 매수일 직후 — 후보는 "그때 알 수 있었던 정보"로만 만든다. 미래 공시가 후보에 새면 사후 확증이 된다)

**처리 파이프라인**
1. lookback 구간 공시 수집 → 이벤트 유형 분류 (4.1과 동일 엔진)
   - **후보 결정성 규칙**: Q2의 공시 기반 후보는 **스크립트(키워드 매핑)로 확정 분류된 공시만** 사용한다. 에이전트 보조 분류는 후보 생성에 개입하지 않는다 — 후보 목록과 후보 ID(r1, r2, ...)가 실행마다 동일해야 컨트랙트 명령(`--choose r1 --dry-run`)이 재현 가능하기 때문이다
2. **질문 세트 구성** (객관식 중심, 기본 2문항 + 선택 1문항):
   - **Q1 — 경로**: "이 종목을 어떻게 알게 되셨나요?" 보기: 유튜브·SNS / 커뮤니티·지인 추천 / 뉴스 기사 / 원래 알던 회사 / 직접 입력. 판정 대상이 아니라 맥락 기록용 — 초보자가 첫 질문부터 막히지 않게 하는 워밍업이기도 하다
   - **Q2 — 이유 (핵심 문항)**: "혹시 이 중에 산 이유가 있었나요? (여러 개 선택 가능)" 보기 = **공시 기반 후보 최대 4개** (매수일에 가까운 순, 각각 근거 공시 rcp_no 보유) + **사유 라이브러리에서 상위 항목** (아래 표 — 투자자들이 실제로 많이 대는 일반 사유) + **직접 입력**. 공시→후보 변환 규칙: 실적 공시(개선)→실적 기대, 자기주식 취득→주주환원 기대, 배당 결정→배당 목적, 신규 시설투자·단일판매공급계약→성장/수주 기대. 유상증자·CB 등 악재성 이벤트는 후보에서 제외한다 (그걸 보고 사는 초보자는 상정하지 않는다)
   - **Q3 — 기대 (선택)**: "얼마나 갖고 갈 생각이었나요?" 단기 / 수개월 / 장기 / 생각 안 해봄 — 이후 점검 주기의 참고 정보
3. **사유 라이브러리** (공시가 없어도 항상 제시되는 일반 사유 — 각 항목은 판정 가능 여부가 미리 정의됨):

   | 라이브러리 보기 문구 | 유형 매핑 | 판정 |
   |---|---|---|
   | "당시 해당 종목의 시장·커뮤니티 분위기나 추천 때문에" | `theme` | 확인불가 + 기대–실체 갭 서술 |
   | "주가가 많이 떨어져서 싸 보여서" | `price_anchor` | 확인불가 + "가격 수준은 공시로 검증되는 사실이 아닙니다" 서술 |
   | "배당 받으려고" | `dividend` | RULE-DIV-01 |
   | "오래 갖고 갈 우량 기업이라고 생각해서" | `earnings_improvement` | RULE-EARN-01 (전사 실적 추세 대조) |
   | "그냥 유명해서 / 잘 모르겠음" | `unknown` | 판정 생략, 타임라인 요약 모드 |

4. **직접 입력 처리** (`type: custom`): 사용자가 자기 말로 이유를 쓰면 원문을 사용자 발화로 원장에 보존하고, 에이전트가 기존 유형으로 분류를 시도한다 — 분류되면 해당 룰로 판정, 분류 불가면 확인불가. 시스템이 사용자의 말을 바꾸지 않고, 판정 근거로도 승격시키지 않는다
   - **분류는 1회 이벤트로 고정한다**: LLM 분류는 실행마다 결과가 달라질 수 있으므로(비결정성), 분류 결과를 원장에 저장한 뒤에는 `thesis-audit`이 절대 재분류하지 않는다 — 판정은 언제나 원장에 저장된 type을 기준으로 하는 결정적 연산이다
   - **분류 결과는 사용자 확인을 거친다**: "이 이유를 '실적 기대'로 이해했어요. 맞나요?" — 사용자가 승인해야 원장에 확정된다. 시스템의 해석을 사용자 모르게 기록하지 않는다
5. **보기 개수 상한과 노출 규칙**: Q2의 총 보기는 **8개 이하** (공시 후보 최대 4 + 라이브러리 최대 3 + 직접 입력 1). 초보자에게 10개짜리 객관식은 또 다른 백지 시험지다. 라이브러리 노출 규칙: `theme`·`price_anchor`·`unknown`은 항상 노출, `dividend`는 해당 종목에 배당 이력이 있을 때만, `earnings_improvement`(우량 기업 보기)는 공시 후보에 실적 항목이 없을 때만 (중복 방지), 그리고 **해당 종목의 주요계정에 영업이익 계정이 존재·유효할 때만** (금융지주 등 영업이익 의미가 불명확한 종목엔 미노출 — G1 리허설에서 확인하는 RULE-EARN 오작동 방지 안전핀)
6. 선택 결과를 클레임으로 변환(5.3의 유형별 클레임 템플릿) → `data/ledger/reasons.json`에 기록 (`source: disclosure | library | user_input` 구분 저장). "잘 모르겠음" 단독 선택 시 `thesis-audit`은 논거 판정 대신 타임라인 요약 모드로 동작함을 안내

**출력**: 원장 기록 + 선택 확인 브리핑 (게이트 통과)

**수용 기준**
- 후보의 근거 공시가 전부 lookback 구간 내의 것 (미래 정보 누출 0건)
- 공시 기반 후보 각각에 근거 공시 rcp_no 저장
- 고정 보기 2개가 항상 존재
- 어떤 선택 조합에서도 원장 기록이 스키마 검증(5.1) 통과

### 4.3 `thesis-audit` — 논거 생사 판정 + 거울 모드

**입력**
- `case`: 케이스 ID (원장에 논거가 기록된 케이스)
- `as_of`: 판정 기준일
- `mirror`: 거울 모드 여부 (기본: false). true이면 현재 손익률(픽스처)을 함께 로드

**처리 파이프라인**
1. 원장에서 논거(클레임) 로드
2. **판정 룰 엔진**(`verdict_engine.py`): 논거 유형별로 사전 정의된 룰 실행 (5.3의 판정 규칙 표). 룰은 MCP 도구로 최신 데이터를 조회해 클레임과 대조하고, `{status, evidence, rule_id}`를 반환
3. 신규 중대 사실 스캔: 논거와 무관하게 매수일 이후 발생한 중대 이벤트(유상증자, CB, 감사의견 변화, 최대주주 변경, 소송 등)를 별도 섹션 "당신이 고르지 않았지만 알아야 할 새 사실"로 첨부
4. `mirror=true`이면: **과거–현재 비교 템플릿**으로 출력한다 — ①과거: 당신이 기록한 매수 이유 (원장 그대로) ②현재: 보유 수익률(시장 대비 병기) + 각 논거의 최신 판정 + 신규 중대 사실 ③마무리: 고정 질문 문구("이 매도의 근거는 가격인가요, 사실인가요?"). 브리핑은 자유 작문이 아니라 **고정 템플릿의 빈칸 채우기**로 생성한다 — 말투가 구조적으로 사실 서술에 갇히게 하는 1차 방어이고, 게이트는 2차 방어다. 고정 질문 문구는 게이트의 허용 목록에 등재된 유일한 의문문 형식
5. 판정 결과를 원장 `audits[]`에 append (판정 이력 누적 — 논거의 생사 변천을 추적 가능) → 브리핑 생성 → 게이트 → 저장

**판정 상태 4종**
- `유효(supported)`: 클레임을 지지하는 최신 공시 데이터 확인
- `약화(weakened)`: 방향은 유지되나 정도가 후퇴 (예: 증익 기대였는데 증익 폭 급감)
- `반증(refuted)`: 클레임과 반대되는 공시 사실 확인
- `확인불가(unverifiable)`: 공시로 검증할 수 없는 유형 (테마 편승 등) — 판정 딱지 대신 "기대–실체 갭" 서술: 그 기대가 회사의 공식 기록(매출·수주 공시)에 잡힌 적이 있는지의 사실만 제시

**출력**: `reports/{case_id}/audit-{as_of}.md` — 판정마다 `{상태} — {근거 공시명, 대조 항목, 수치 before/after} [원문]({url}) (rule: {rule_id})`

**수용 기준**
- 모든 판정에 rule_id와 원문 URL 존재 (재현 가능성)
- 같은 원장 + 같은 `as_of` → 항상 같은 판정 (결정성 테스트)
- `확인불가` 유형에 유효/반증 딱지가 절대 붙지 않음
- 거울 모드 출력에 행동 지시·예측 표현 0건 (게이트 검증)

### 5.3 논거 유형 카탈로그와 판정 규칙 (판정의 심장)

**상황과 고민**: "논거가 아직 유효한가"를 자유 텍스트로 판정하게 하면 LLM의 주관이 개입하고 재현이 불가능하다. **해결**: 논거는 애초에 객관식(재인)으로 수집되므로, 유형이 유한하다. 유형마다 판정 룰을 코드로 정의한다.

| 유형 (type) | 후보 생성 트리거 (매수일 주변 공시) | 클레임 템플릿 | 판정 룰 (rule_id) |
|---|---|---|---|
| `earnings_improvement` 실적 기대 | 실적 관련 정기·잠정 공시 중 개선 방향 | "**전사** 영업이익이 개선 추세다" | RULE-EARN-01: 최신 분기 **전사** 영업이익 YoY — 개선=유효 / +이나 증가폭 50%↓ 축소=약화 / 감익·적자전환=반증. **① 재무제표 기준 고정: `fs_div`는 연결(CFS)로 고정하고, 연결재무제표가 없는 종목만 별도(OFS)로 폴백하되 브리핑에 어느 기준인지 표기한다. ② 동일 보고서 종류끼리 YoY: 1분기는 전년 1분기와, 3분기 누적은 전년 3분기 누적과 비교한다 — 손익 계정이 누적 기준이라 다른 보고서끼리 비교하면 기간이 어긋나 틀린 부호가 나온다. ③ 부문별(세그먼트) 손익은 OpenDART 주요계정 API가 제공하지 않으므로 판정 대상은 전사 지표로 한정하고, 브리핑 서술에서도 부문 단위 단정을 하지 않는다** |
| `shareholder_return` 주주환원 기대 | 자기주식 취득 결정 | "회사가 자기주식 취득 등 주주환원을 실행한다" | RULE-RET-01: 취득 완료·신규 결정=유효 / 중단·처분=반증 |
| `dividend` 배당 목적 | 배당 결정 공시 | "배당이 유지 또는 확대된다" | RULE-DIV-01: 주당배당금 유지·증가=유효 / 감소=약화 / 무배당 전환=반증. **비교는 동일 주기끼리 한다** (분기배당↔직전 분기배당, 연간↔직전 연간) — 주기가 다른 값을 비교하면 거짓 감소가 나온다 |
| `growth_order` 성장·수주 기대 | 신규 시설투자, 단일판매·공급계약 | "발표된 투자/계약이 정상 진행된다" | RULE-GRW-01: 후속 정정 없음=유효 / 규모 축소 정정=약화 / 해지·중단 공시=반증 |
| `new_business` 신사업 기대 | 사업목적 추가, 신사업 관련 주요사항 | "신사업이 공식 기록상 진행된다" | RULE-NEW-01: 관련 후속 공시 존재=유효 / 장기 무소식=약화(기간 파라미터) / 중단 공시=반증 |
| `theme` 테마·분위기 편승 (라이브러리) | 없음 (비공시 재료) | 판정 대상 아님 | RULE-THM-01: **확인불가** 고정 + 기대–실체 갭 서술 (관련 매출·수주 공시 존재 여부만 사실 제시). **주의: 뉴스를 쓰지 않으므로 시스템이 구체 테마명을 생성·명명하지 않는다.** 보기 문구는 "당시 해당 종목의 시장·커뮤니티 분위기나 추천 때문에"로 하고, 사용자가 테마명을 직접 말하면 사용자 발화로만 기록한다 (판정 근거 아님) |
| `price_anchor` 저가 매수 (라이브러리) | 없음 (가격 재료) | 판정 대상 아님 | RULE-PRC-01: **확인불가** 고정 + "가격 수준은 공시로 검증되는 사실이 아닙니다" 서술. 보유 수익률·시장 대비 수치는 참고 사실로만 병기 |
| `custom` 직접 입력 | 사용자 자유 텍스트 | 분류 시도 후 결정 | 기존 유형으로 분류 성공 시 해당 룰 적용, 실패 시 확인불가. 사용자 원문은 발화 그대로 보존하며 시스템이 바꿔 쓰지 않는다 |
| `unknown` 잘 모르겠음 (라이브러리) | 없음 | 없음 | 판정 생략, 타임라인 요약 모드 |

- 룰 엔진은 위 표를 코드(`verdict_engine.py`의 룰 레지스트리)로 구현하며, 표와 코드의 일치가 테스트 대상이다
- **약화(weakened)의 경계값**(예: 증가폭 50% 축소)은 데모용 초기값이며, 브리핑에는 경계값 자체를 함께 표기해 판정이 자의적이지 않음을 보인다

### 5.4 데모 케이스 구성 (아엔 결정: 전 케이스 실존 대형주)

| 케이스 | 종목 | 목적 | 데이터 |
|---|---|---|---|
| case-001 | 삼성전자 (확정 제안) | 타임라인 + 유효 판정 + 수익률·시장 대비 표기 + "하락했지만 나빠진 사실 0건" 장면 | 실제 공시 스냅샷 |
| case-002 | **카카오 (아엔 확정)** | 급변–공시 매칭 + 약화/반증 판정 + 거울 모드. 카카오는 실제 변동성 이벤트와 공시가 풍부하고, **출제사의 모기업 주식이라 "본인들 주식으로 검증했다"는 데모 내러티브가 성립** | 실제 공시 스냅샷 |
| case-003 | 배당주 1종 — **후보 우선순위: ①KB금융(분기배당 도입·배당 공시의 규칙성) ②SK텔레콤(고배당·공시 단순) ③KT&G(장기 배당 이력)**. 픽스처 수집 게이트에서 세 후보의 실제 배당 공시 데이터를 대조해 Claude가 최종 1종을 제안하고 아엔이 승인한다 (아엔 위임 결정) | 배당 논거 유효/약화 케이스 | 실제 공시 스냅샷 |

- **부정 판정의 리스크 관리**: "반증됨"은 회사에 대한 평가가 아니라 **사용자의 논거 대비 공시 사실의 판정**이다. 브리핑에 이 정의를 고정 문구로 명시하고, 판정 근거는 전부 해당 회사가 스스로 제출한 공시 원문 인용으로만 구성한다 — 사실 인용에는 평판·법적 문제가 성립하지 않는 구조
- 실데이터 수집 후 **케이스 리허설**(각 케이스가 의도한 장면을 실제로 만들어내는지 확인)을 개발 게이트 최전방에 배치한다

## 7. 컴플라이언스 게이트 명세 (`tools/compliance_gate.py`)

**상황**: 원칙 2(해석·권유 금지)를 "주의하겠다"는 약속으로 두면 증명이 안 된다. **해결**: 모든 브리핑이 저장 전 반드시 통과해야 하는 기계 검사.

### 7.1 검사 항목

1. **금지 패턴 (fail)**: 행동 지시어(사세요/파세요/매수하세요/매도하세요/버티세요/손절·익절 권고형), 예측 표현(반등할/오를/하락할 것으로 예상/전망됩니다), 가치 평가어(저평가/고평가/싸다/비싸다/기회/유망), 단정 표현(확실/분명히 오른다). 정규식 사전은 `tools/gate_rules.yaml`로 외재화 — 규칙 추가·검토가 코드 수정 없이 가능
   - **오탐 방지 원칙**: 패턴은 지시형·권고형·예측형 **종결 표현**을 겨냥한다. "매수", "매도" 같은 명사 단독이나 사실 서술("임원이 주식을 매도했다는 공시", "매수 논거")은 검출 대상이 아니다. 게이트 양성 테스트 세트에 이런 사실 서술 문장을 포함해 오탐 0건을 함께 검증한다
2. **필수 요소 (fail if missing)**: 데이터 기준일 문구, 고정 면책 문구, 판정 항목당 원문 URL(마크다운 링크 존재 검사), "반증은 회사 평가가 아니라 논거 대비 사실 판정" 정의 문구(판정 브리핑에 한함)
   - **판정 상태별 차등**: 유효/약화/반증 판정 라인은 원문 URL 필수. **확인불가·데이터 부재 판정 라인은 인용할 원문이 존재하지 않으므로**, URL 대신 "대조를 시도한 데이터 소스와 부재 사실" 문구를 필수로 한다 — 링크를 일률 강제하면 부재의 정직(원칙 3)과 게이트가 서로 충돌한다
   - **허용/금지의 경계**: "시장 대비 상회/하회", "+32%", "전기 대비 감소" 같은 **산술·사실 비교는 허용**된다 (계산 결과). "좋다/나쁘다/선방/부진/양호" 같은 **평가어는 금지**된다 (판단). 게이트 규칙 사전에 이 경계를 주석으로 명시한다
3. **허용 예외 목록**: 거울 모드의 고정 질문 문구 등 사전 등재 문장만 예외 처리 (예외는 파일에 명시돼 심사 시 검토 가능)

### 7.2 출력

- exit 0 (통과) / exit 1 (위반) + 위반 상세(패턴, 위치) 리포트
- 모든 실행 로그를 `reports/{case_id}/gate-log-{timestamp}.txt`로 남긴다 — 데모의 "통과 로그 공개" 장면의 원천

### 7.3 게이트 자체의 테스트

- 양성 세트(정상 브리핑 샘플 — 사실 서술로서의 "매수/매도" 단어 포함)와 음성 세트(금지 문장 삽입 샘플)를 `tests/gate/`에 두고, 게이트가 음성 세트를 전부 잡고 양성 세트를 통과시키는지 검사 — "게이트를 검증하는 테스트"까지 있어야 "코드로 증명"이 완성된다

### 스킬 컨트랙트 (정적 검증 폴백의 원천)

각 SKILL.md 말미에 "이 스킬이 정상이라면 이 명령이 exit 0"인 컨트랙트 명령을 명시한다 (아래의 날짜·케이스 값은 자리표시자이며, G1 리허설 확정값으로 치환된다):

- `reason-recall`: `python -m tools.run_skill recall --case case-001 --choose r1 --dry-run` (선택 시뮬레이션) `&& python tools/validate_ledger.py`

Codex 라이브 실행이 막힌 환경에서도 이 명령들의 exit 0 + 산출물 동봉 + MCP stdio 기동 확인으로 품질을 증명한다. README에 실행 환경 제약을 정직하게 고지한다.

## 9. 설계 결정 기록 — 고민과 트레이드오프

### 9.1 판정을 LLM에 맡길까, 룰로 짤까

- **고민**: LLM 판정은 구현이 빠르고 유연하지만, 같은 입력에 다른 답이 나올 수 있고 "왜 유효인가"를 증명할 수 없다. 룰은 개발 비용이 들고 커버리지가 좁다
- **결정**: 룰. 근거 셋 — ①객관식 수집 덕에 논거 유형이 유한해서 룰이 성립한다 (reason-recall이 thesis-audit의 성립 조건이라는 구조적 발견) ②재현성은 이 제품의 존재 이유(검증 가능한 근거)와 직결 ③"그게 맞다는 것을 스스로 증명할 수 있는" 인재상·평가 기준에 룰+테스트가 정확히 부합
- **버린 것**: 자유 텍스트 논거의 판정. 로드맵으로만 남긴다 (자유 입력 → 유형 분류 → 기존 룰 재사용의 경로가 인터페이스상 열려 있음)

### 9.2 후보 생성의 미래 정보 누출 문제

- **고민**: reason-recall이 매수일 "이후" 공시로 후보를 만들면, 사용자가 그때 알 수 없었던 정보로 논거를 소급 조작하게 된다 (사후 확증 편향을 도구가 조장)
- **결정**: 후보 수집 창을 D-30 ~ D+3으로 제한하고, `as_of`를 매수일 직후로 고정. 테스트(`test_recall.py`)가 누출 0건을 검사. D+3을 허용한 이유: 매수 직후 며칠 안의 공시는 "산 직후 알게 된 이유"로 현실적이기 때문 — 이 경계값도 브리핑에 명시

### 9.3 픽스처 종목: 실존 vs 가상 (아엔 결정: 전 케이스 실존)

- **고민**: 실존 종목 실데이터는 원문 링크가 실제로 열려 심사 설득력이 최강이지만, "반증됨" 딱지를 실존 기업에 박제하는 리스크가 있다. 초기에 하이브리드(긍정=실존, 부정=가상)를 제안했다
- **결정 (아엔)**: 전 케이스 실존 대형주로 확정. 하이브리드 기각 — 가짜 링크가 하나라도 섞이면 "링크를 눌러 검증하라"는 데모의 힘이 죽는다
- **리스크 관리**: 부정 판정도 해당 회사가 스스로 제출한 공시 원문 인용으로만 구성하고, "반증은 회사 평가가 아니라 사용자 논거 대비 사실 판정"이라는 정의 문구를 브리핑에 고정한다 (게이트 필수 요소로 강제)

### 9.4 가격 데이터의 위치

- **고민**: 가격을 아예 배제하면 철학은 순수해지지만 S2(급락일의 공포)라는 가장 흔한 상황을 못 다룬다. 가격을 판정에 쓰면 규제 선을 넘는다
- **결정**: 가격은 "어느 날짜를 들여다볼지"의 좌표(x축)로만. 판정 룰 엔진의 입력에서 가격을 구조적으로 배제 (verdict_engine은 price 도구에 접근하지 않는다 — 코드 수준 격리)

### 9.5 "약화(weakened)" 상태를 둘 것인가

- **고민**: 유효/반증 이분법이 단순하고 안전하지만, 현실의 실적은 "좋아지긴 했는데 기대만큼은 아님"이 흔하다. 이분법은 이 회색지대를 거짓 유효로 뭉갠다
- **결정**: 약화를 셋째 상태로 도입하되, 경계값을 룰에 하드코딩하고 브리핑에 경계값을 노출한다. 자의적 판단으로 보이지 않게 하는 장치

### 9.6 테마 논거의 처리

- **고민**: "테마 때문에 샀다"는 가장 흔한 실제 이유인데 공시로 검증 불가. 보기에서 빼면 현실 부정, 넣고 판정하면 원칙 위반
- **결정**: 라이브러리 보기로 유지 + 판정 상태 `확인불가` 신설 + 기대–실체 갭 서술(관련 매출·수주 공시의 존재 여부만 사실 제시). "확인 불가함을 확인해주는 것"도 부재의 정직의 일부

### 9.7 게이트를 프롬프트로 할까, 코드로 할까

- **고민**: 프롬프트 지시("권유 표현 쓰지 마")는 비용 제로지만 증명 불가·우회 가능. 코드 게이트는 정규식 사전의 유지 비용과 오탐(정상 문장 차단) 위험
- **결정**: 코드 게이트 + 규칙 외재화(yaml) + 게이트 자체를 검증하는 음성 테스트 세트. 오탐은 지시형·권고형 종결 표현만 겨냥하는 패턴 설계와 허용 예외 목록으로 관리. "규제 준수를 코드로 증명"이 이 제품의 심사 차별점이므로 비용을 감수

### 9.8 해외주식: 스텁조차 넣지 않는다 (아엔 결정)

- **고민**: 어댑터 스텁을 남기면 "확장 가능성"을 보여줄 수 있지만, 동작하지 않는 코드가 저장소에 존재하는 것 자체가 완성도 인상을 해치고 스코프를 흐린다
- **결정**: 국내 전용으로 확정 ("아직 다루지 않는다"). 스텁·인터페이스 추상화 없이 DART 클라이언트를 단순 구현하고, README에 국내 전용임을 한 줄로 명시. 하지 않는 것을 하는 척하지 않는다 — 부재의 정직은 스코프에도 적용된다

### 9.9 수익률 태그: "좋음/나쁨"이 아니라 "시장 대비" (아엔 요구의 변형 수용)

- **요구**: 보유 수익률을 보여주고 한눈에 알 수 있는 태그를 달자
- **고민**: "좋음/보통/나쁨"은 가치 평가어라서 우리 게이트가 스스로 차단하는 표현이고, 규제 선(단정적 판단 제공)에도 닿는다. 그런데 수익률 숫자만 던지면 초보자는 그게 좋은 건지 감이 없다 — 요구의 본질(직관성)은 정당하다
- **결정**: 수익률은 항상 표시하되, 태그는 **같은 기간 시장 지수와의 산술 비교**(상회/유사/하회)로 단다. "-12%인데 시장은 -15%"라면 "시장 대비 상회" — 이건 계산 결과(사실)이지 판단이 아니라서 게이트를 통과하고, 초보자에게는 "내 종목만 떨어진 게 아니구나"라는, '좋음' 태그보다 더 유용한 맥락을 준다
- **격리 유지**: 수익률·시장 대비 태그의 계산은 브리핑 조립 계층(`run_skill`)에서 하며, 판정 룰 엔진(`verdict_engine`)의 입력으로는 절대 들어가지 않는다 — 9.4의 가격 격리 원칙과 충돌하지 않음을 명시해 둔다

### 9.10 논거 수집을 단일 객관식에서 질문 세트로 확장 (아엔 결정)

- **요구**: 질문 하나가 아니라 여러 개, 보기도 그럴싸한 여러 개, 직접 입력도 허용
- **설계**: Q1(알게 된 경로 — 워밍업·맥락) / Q2(이유 — 공시 후보 + 사유 라이브러리 + 직접 입력) / Q3(보유 기대 — 선택). 사유 라이브러리는 항목마다 판정 가능 여부를 미리 정의해 "보기 확장"이 "판정 불능 확장"이 되지 않게 했다. 직접 입력은 유형 분류를 시도하고(1회 고정 + 사용자 확인), 분류되면 룰 재사용·안 되면 확인불가 — 9.1에서 로드맵으로 미뤘던 자유 입력이 이 경로로 스코프에 안전하게 들어왔다

## 10. 알려진 제약과 대응 (착수 전 정직 고지)

1. **픽스처 수집은 아엔의 로컬에서만 가능하다**: 개발 보조 환경(샌드박스)은 외부 네트워크가 패키지 저장소로 제한되어 OpenDART·KRX에 접근할 수 없다. `tools/fetch_dart.py`는 아엔 로컬에서 실행하는 것을 전제로 작성하며, 수집된 픽스처 JSON을 저장소에 커밋한 뒤부터는 어느 환경에서도 오프라인 개발·데모가 가능하다
2. **시나리오의 예시 수치는 가상이다**: "-12%", "영업이익 +32%", "4월 30일 분기보고서" 등은 컨셉 설명용 가상 수치다. 실존 종목 픽스처를 만들면 브리핑의 날짜·수치는 실제 공시를 따른다. 특히 1분기 보고서의 법정 제출 기한은 분기말 후 45일(5월 중순)이므로, 실데이터에서는 시나리오 예시와 날짜가 달라진다 — 데모 리허설 때 시나리오 문서의 예시를 실데이터 기준으로 갱신할지 여부를 결정한다
3. **corp_code 사전 다운로드**: OpenDART는 종목코드가 아닌 자체 고유번호(corp_code)를 쓰며, 전체 매핑 파일(corpCode.xml zip)을 최초 1회 내려받아야 한다. `fetch_dart.py`가 처리하고 픽스처에 매핑을 포함한다
4. **정정공시 매칭의 한계**: RULE-GRW-01(수주·투자의 후속 정정 추적)은 원 공시와 정정 공시를 제목·유형 휴리스틱으로 연결한다. 완전한 계보 추적은 데모 스코프를 넘으므로, 판정 근거에 매칭 방식을 명시해 한계를 숨기지 않는다
5. **대형주 급변일의 희소성**: 삼성전자급 대형주는 일간 ±5% 변동이 드물어 case-001에서 급변일 매칭 장면이 비어 있을 수 있다. 이는 결함이 아니라 "매칭할 급변이 없었음"이라는 정직한 출력이며, 급변–공시 매칭과 거울 장면의 시연은 변동성 이벤트가 풍부한 case-002(카카오 제안)가 전담한다. 픽스처 수집 직후 케이스 리허설에서 각 케이스가 의도한 장면을 실제로 만들어내는지 최우선 확인한다
6. **분기 재무 데이터의 시차**: `as_of`를 분기보고서 제출 전 날짜로 잡으면 RULE-EARN-01이 대조할 최신 분기가 없을 수 있다. 이 경우 판정을 건너뛰지 않고 "직전 판정 유지 + 최신 데이터 부재 고지"로 출력한다 (부재의 정직)
7. **케이스의 매수일은 페르소나 설정값이다**: 데모 케이스의 `buy_date`·`as_of`는 실데이터에서 의도한 장면(유효 판정, 급변–공시 매칭, 반증)이 성립하도록 선정한다. 이것은 데이터 조작이 아니다 — 공시·시세 데이터는 손대지 않고, 가상 인물 A씨가 "언제 샀는지"라는 설정만 고르는 것이며, 어떤 날짜를 골라도 출력되는 사실은 전부 실제 공시 원문이다. 케이스 리허설의 첫 작업이 바로 이 날짜 탐색이다
8. **반증 장면이 실데이터에서 안 나올 리스크**: case-002(카카오)에서 "논거 반증" 장면이 성립하려면 [매수일 주변에 긍정 재료 공시 존재 → 이후 그 재료가 공시로 뒤집힘]의 실제 조합이 필요하다. 리허설에서 이 조합을 찾지 못하면 폴백 순서로 대응한다: ①"약화" 판정 장면으로 대체 (실적 증가폭 축소는 흔해서 성립 확률 높음) ②buy_date·as_of 재탐색 ③반증 시연을 case-003으로 이관 (배당 축소 사례). 어떤 폴백에서도 데이터는 손대지 않는다

## 13. 부록 — 한 사이클로 보는 동작 흐름 (쉬운 설명)

실무자 김OO가 Codex를 열고 왜샀지를 쓰는 하루를 그대로 따라간다.

**준비 (이미 끝나 있는 상태)**: 아엔이 로컬에서 수집 스크립트를 한 번 돌려서, DART에서 받은 공시·재무·시세가 전부 `fixtures/` 폴더에 파일로 얼려져 있다. 이후로는 인터넷이 필요 없다.

**① "case-001 타임라인 보여줘"** — Codex가 buy-timeline 스킬의 지시서(SKILL.md)를 읽고 파이프라인을 실행한다. MCP 서버(우리가 만든 작은 심부름꾼 프로그램)에게 "삼성전자, 매수일부터 기준일까지 공시 목록 줘"라고 시키면, 심부름꾼은 얼려둔 파일에서 **기준일까지 제출된 것만** 골라 준다(시간 여행 필터). 분류기가 수백 건 중 중요한 것만 추리고("그 외 N건"은 접고), 시세 파일에서 수익률과 시장 대비를 계산하고, 템플릿의 빈칸에 끼워 넣는다.

**② 게이트 통과** — 완성된 브리핑이 저장되기 직전, 검사기가 훑는다: 권유하는 말투 없나? 모든 판정에 원문 링크 있나? 면책 문구 있나? 통과하면 exit 0과 함께 `reports/` 폴더에 저장되고 통과 로그가 남는다. 걸리면 저장 자체가 거부된다.

**③ "왜 샀는지 물어봐 줘"** — reason-recall이 매수일 주변 공시에서 후보를 만들고, 사유 라이브러리 보기를 섞어 객관식을 내민다. 사용자가 고르면 그 선택이 논거 원장(`reasons.json`)에 기록된다 — 스키마 검사를 통과해야만 기록된다.

**④ 시간이 흐른 뒤 — 1주일이든 1년이든 (실제로는 기준일 파라미터만 바꿔서)** — thesis-audit이 원장에서 논거를 꺼내 판정 룰 엔진에 넣는다. 룰 엔진은 "실적 기대라고? 그럼 최신 분기 영업이익을 작년 같은 분기와 비교해 봐"처럼 **유형별로 정해진 계산**만 한다. AI의 감이 아니라 뺄셈과 비교라서, 백 번 돌려도 같은 답이 나온다. 결과가 "유효 — 근거 공시, 대조 항목, 규칙 번호, 원문 링크"로 브리핑에 박히고, 다시 게이트를 거쳐 저장된다.

**⑤ 검증** — 김OO는 브리핑의 링크를 눌러 DART 원문과 대조하고, 게이트 로그를 열어 보고, 재현 명령을 직접 돌려 같은 결과가 나오는 것을 확인한다 — "이건 출시 검토 가능하겠다."

역할 분담을 한 줄로 요약하면: **Codex(AI)는 말을 다듬는 역할만, 숫자 계산과 판정은 전부 일반 프로그램이, 마지막 검문은 게이트가.** AI가 틀릴 수 있는 자리를 구조적으로 없앤 배치다.
